harness-claude-code · evolution note

Agent-team mode → Truth-on-Git mode

How this plugin moved from a single long-running orchestrator that held an in-memory agent team and drove an issue to a merged PR in one session, to a set of stateless pickup commands that read and write GitHub labels and dispatch one-shot agents — recoverable, loopable, and unattended.

Where the change happened

v0.2.0 — v0.4.0
Agent-team mode
One implement-issue skill drives a worktree end-to-end. After PR open it spins up a persistent Team of named engineer + reviewer members that SendMessage each other until both review gates are clean.
v0.5.x — v0.8.x
Transition
Reviewers and the engineer are rewritten to read state from GitHub and flip labels themselves. create-issues attaches a slice branch to each slice issue. The first pickup-* commands appear.
v0.10.0 — v0.16.x
Truth-on-Git, PR-scoped
The orchestrator is gone. Three pickup commands plus a PreToolUse pre-push hook fully replace it. Every agent is anonymous, one-shot, mode=auto; review:* labels live on the slice PR.
v0.17.x — v0.22.x
Truth-on-Git, task-scoped
review:* moves off the slice PR and onto each task issue. The engineer gains three modes (A/B/C). Seven pickup/close commands split the lifecycle into finer phases; the e2e gate is now owned by a GitHub Actions workflow on the slice PR.
v0.23.x — current (v0.27.0)
Lifecycle skills + orchestrator
The seven pickup/close commands become eight auto-loadable skills; a new create-draft-pr skill takes PR creation away from e2e-author; a single /implement-feature command chains all eight skills, and /loop /implement-feature drives the whole lifecycle unattended.

Two mental models, side by side

Agent-team mode v0.4.0

A single human-launched session calls the implement-issue skill. That session is the workflow — kill it and the work stops.

What the orchestrator owns in memory

  • The feature worktree path.
  • A TaskCreate task list mirroring the sub-issues, with blockBy edges translated from Blocked by.
  • A persistent Team of four named members: backend-engineer, frontend-engineer, security-reviewer, code-reviewer — handing work to each other via SendMessage.

How a fix round flows

orchestrator
  └─ SendMessage → security-reviewer
                     └─ SendMessage → backend-engineer (fix)
                     └─ re-validate → orchestrator
  └─ SendMessage → code-reviewer
                     └─ SendMessage → frontend-engineer (fix)
                     └─ re-validate → orchestrator
TeamDelete + worktree teardown

Costs

  • If the session dies mid-flight, recovery means restarting and re-discovering progress from scratch.
  • Cannot run unattended in a loop — the orchestrator is the loop.
  • The whole flow lives inside one mega-skill.

Truth-on-Git mode (PR-scoped) v0.10.0

No orchestrator session. The state of the world is the set of GitHub labels on issues and PRs, plus the slice branch attached to each slice issue. Three pickup commands read that state, flip a label, and dispatch one one-shot agent per unit of work.

What lives where

  • Issue labels: level:slice|task, status:draft|ready-to-implement|in-progress|reviewed, kind:feature, type:e2e|backend|frontend.
  • PR labels: review:code-{pending,running,passed,need-fix}, review:security-{…}, review:e2e-{…} (owned by GitHub Actions).
  • Slice branch attached to the slice issue at creation time via gh issue develop.

How a fix round flows

/loop /pickup-pr-for-review
  └─ flips review:code-pending → -running
  └─ Agent code-reviewer (one-shot)
       └─ posts findings, sets -need-fix, exits

/loop /pickup-reviewed-pr
  └─ sees -need-fix
  └─ Agent engineer Mode B (one-shot)
       └─ pushes fix, sets -pending, exits

Wins

  • Crash anywhere — the next pickup pass picks up from labels.
  • /loop /pickup-* runs unattended.
  • Each command is small enough to read top-to-bottom.

The three pickup commands at a glance — v0.10.0 lineup

/pickup-task-for-implementation

finds → dispatches → writes
  • Promotes ready-and-unblocked slice issues to in-progress.
  • For each task with level:task + status:ready-to-implement + kind:feature and zero open Blocked by:
    • type:e2e → e2e-author
    • type:backend|frontend → engineer Mode A
  • When the slice's last task closes, the engineer that closed it opens the slice PR's review:*-pending labels.

/pickup-pr-for-review

finds → dispatches → writes
  • Scans draft PRs for any review:<gate>-pending.
  • Flips each to -running and dispatches the matching one-shot reviewer (code → code-reviewer; security → security-reviewer).
  • The reviewer posts findings and flips to -passed or -need-fix. The e2e gate is owned by GitHub Actions.

/pickup-reviewed-pr

finds → dispatches → writes
  • Promotes and squash-merges PRs whose code, security, and e2e gates are all green.
  • Reverts conflicted PRs back to draft.
  • Dispatches one-shot fullstack engineer Mode B for any PR carrying -need-fix or failing CI, with the scenario list (conflict/review/ci).

Why this redesign

The agent-team mode worked but was fundamentally a session. Every failure mode — orchestrator timeout, network hiccup mid-team-handoff, user-side restart — required reconstructing state from whatever made it onto the PR. The per-session TeamCreate overhead also bound one merge to one continuous Claude Code conversation.

Truth-on-git mode treats GitHub the way every other CI/CD pipeline does: as the durable, observable substrate. Labels are the queue; the slice branch is the workspace; the PR is the audit log. Each agent is a small function that reads a label, mutates the world, writes a label, exits. The harness can crash, the laptop can sleep, the loop can be interrupted — none of it loses progress.

Iterating: PR-scoped → task-scoped reviews (v0.10.0 → v0.22.0)

The v0.10.0 truth-on-git design parked every review label on the slice PR. That worked when each slice was one or two commits, but as slices grew into e2e + backend + frontend trios, one verdict per PR meant a single typo in the frontend task would re-block the backend review that had already passed. The reviewer's findings comment also drifted away from the task that produced them, making it harder for the next fix round to know which task to re-open.

The fix was to move the entire review:* family off the slice PR and onto each level:task issue, give each task its own independent code and security gates, and split the original three pickup commands into seven smaller ones — each owning one phase of the lifecycle. The engineer agent grew a second and then a third mode to match the new dispatch points; the e2e label family was retired in favor of a GitHub Actions workflow check on the slice PR.

Current pickup lineup (v0.22.0)

/pickup-slice-for-implement

slice → in-progress · tasks unlocked
  • Picks level:slice + status:ready-to-implement + kind:feature with zero open Blocked by.
  • Flips slice to status:in-progress; appends status:ready-to-implement to every level:task + kind:feature sub-issue.
  • Mutates labels only — never checks out, edits, or pushes.

/pickup-task-for-implement

task → in-progress · agent dispatched
  • Picks ready, unblocked tasks one at a time.
  • type:e2e → e2e-author; type:backend / type:frontend → engineer Mode A.
  • Agent pushes commits to the slice branch and adds review:code-pending (+ review:security-pending if not e2e) onto the task.

/pickup-task-for-review

review:*-pending → -running
  • Scans in-progress tasks with review:code-pending or review:security-pending.
  • Flips just the pending gate to -running; dispatches code-reviewer or security-reviewer.
  • Reviewer posts a structured comment on the task and flips to -passed or -need-fix.

/pickup-reviewed-task-for-fix

strip-lock · Mode C dispatch
  • Picks tasks carrying review:*-need-fix with no in-flight gate.
  • Strips every review:{code,security}-{need-fix,passed} — absence is the lock.
  • type:backend / type:frontend → engineer Mode C (reads user directives first, then reviewer findings); type:e2e → e2e-author. Agent re-adds review:*-pending after pushing.

/pickup-failed-pr-for-fix

status:fix-in-progress lock
  • Scans draft PRs for failing CI checks and/or CONFLICTING mergeability.
  • Locks each affected PR with status:fix-in-progress; dispatches engineer Mode B with scenarios {conflict, ci}.
  • Does not touch review:* (those live on tasks) and does not merge.

/close-task-issue

task gates green → closed
  • Closes status:in-progress tasks whose required review gates all reached -passed.
  • backend / frontend: code-passed + security-passed. e2e: code-passed only.
  • Strips status:in-progress and closes with --reason completed. Skips any task with a fresh review cycle in flight.

/close-pr

MERGEABLE + green CI → squash-merge
  • Promotes MERGEABLE draft PRs with a fully green statusCheckRollup and squash-merges them with --delete-branch.
  • Strips status:in-progress from the linked slice and closes the slice issue.
  • Reviews are out of scope — they live on task issues.

hooks/engineer-pre-push.sh

PreToolUse · Bash gate
  • Fires on every Bash call, no-ops unless the command contains git push and the cwd is under /tmp/git-worktree/.
  • First push of the slice: narrows to the active task's stack via the most recent Refs #<n> trailer. Once a PR exists: runs both stacks.
  • backend = ruff / mypy / bandit / pytest; frontend = biome / tsc --noEmit / npm audit / jest. Failure denies the push so the engineer retries.

Iterating: pickup commands → lifecycle skills (v0.22.0 → v0.27.0)

The v0.22.0 lineup worked, but each pickup phase lived as a command: discoverable only if the user remembered to type /pickup-…, and not reachable via the skill auto-load path that tdd-workflow, git-workflow, etc. all relied on. Two other rough edges were carried over from the truth-on-git cut: e2e-author doubled as a PR opener (its first push of the slice would create the draft PR as a side-effect), and end-to-end shipping still required a human typing each pickup command in turn — even though every phase was already idempotent and self-skipping.

The v0.27.0 cut converts the seven pickup/close commands into auto-loadable skills under skills/, pulls draft PR creation out of e2e-author into a dedicated create-draft-pr skill (so PR creation has a single owner, gated on every task being closed), and adds a single thin orchestrator command /implement-feature that calls all eight skills in order. Each skill stays idempotent and self-skipping, so wrapping with /loop /implement-feature drives the whole slice from "ready" to "merged" with no human in the chain.

Current lifecycle lineup (v0.27.0)

Eight skills, in the order /implement-feature calls them on every pass. Each is idempotent — fires only on items matching its filter, reports "nothing to pick up" otherwise — so a single /implement-feature pass safely sweeps whatever state is ready, and /loop /implement-feature keeps advancing until every slice has merged.

The orchestrator: /implement-feature

A single thin command, commands/implement-feature.md, chains the eight skills above in order, sequentially. It does not wait for backgrounded sub-agents (engineers, e2e-authors, reviewers) to finish; once a skill dispatches its agents, the command moves on. To keep the lifecycle advancing end-to-end, wrap with /loop so subsequent passes catch state that lands after backgrounded work completes. 

/loop /implement-feature
     │
     ▼   (one pass = one call per skill, in this order)
  kickoff-slice-issue   ──►  ready slices flipped to in-progress
                              task sub-issues primed with
                              status:ready-to-implement
  implement-task-issue  ──►  ready tasks dispatched
                              (engineer Mode A / e2e-author, backgrounded)
  review-task-issue     ──►  pending gates flipped to -running
                              (code-reviewer / security-reviewer, backgrounded)
  fix-task-issue        ──►  -need-fix tasks dispatched
                              (engineer Mode C / e2e-author fix, backgrounded)
  close-task-issue      ──►  tasks whose required gates are all
                              -passed → closed
  create-draft-pr       ──►  slices whose tasks have all closed
                              → draft PR opened, Closes #slice + #tasks
  fix-pr                ──►  draft PRs with CI red / CONFLICTING
                              locked + engineer Mode B dispatched
  close-pr              ──►  MERGEABLE + green draft PRs → squash-merge
                              + slice closed

  summary line: implement-feature: pass complete (kickoff X / implement X /
                review X / fix-task X / close-task X / draft-pr X /
                fix-pr X / close-pr X)

Order is load-bearing: each step's filter depends on labels mutated by an earlier step (e.g. implement-task-issue can't see a task until kickoff-slice-issue has primed it; create-draft-pr can't fire until close-task-issue has closed every task under the slice). The orchestrator never reorders, never skips, never parallelizes.
